阶段边界

目标是判断当前雏形能否作为“可落地应用”的下一阶段基础，而不是要求它已经达到 Pika 级实时音视频能力。 验收语言必须明确区分“已实现”和“下一阶段目标”。

已纳入验收 观察项 不通过即阻断

本阶段明确不承诺

  • 不验收真正 WebRTC / Realtime API 双向音频。
  • 不验收多人连麦、服务端房间、云端部署和账号体系。
  • 不验收训练数据闭环学习，只验收启发式准实时反馈是否稳定可解释。
  • 不验收商业级语音克隆，只验收当前 TTS voice 是否适配 avatar 形象。

测试流水线

01 静态健康

npm run typecheck 和 npm test 必须通过。

02 API 冒烟

确认 chat / tts 能返回；TTS voice id 能合成 MP3。

03 桌面视觉

Chrome 打开 /debate，检查首屏无遮挡、无错误 overlay。

04 移动端视觉

390×844 或同级窄屏，检查 Training 抽屉和输入区。

05 通话链路

文字输入、麦克风输入、AI 回复、TTS、口型、中断全部走一遍。

06 资源观察

10 分钟连续使用，观察 CPU、温度、卡顿、风扇和内存。

验收门槛

P0

硬性阻断

  • 页面空白、Next/Vite 错误 overlay、核心按钮不可点击。
  • 语音/文本任一路径无法完成一轮对话。
  • Stop / End Call 无法中断 LLM 或 TTS。
  • 移动端或桌面首屏遮挡 avatar 脸、输入框或主要控制。
  • 连续测试导致温度或 CPU 明显失控。
P1

体验必须修

  • TTS voice 与当前 avatar 明显违和。
  • 训练反馈文案无法解释状态变化。
  • 移动端抽屉遮住内容但无法关闭或滚动。
  • 明显的布局抖动、长文本溢出、按钮文字挤压。
P2

可延期优化

  • 非核心动效不够精致。
  • Live2D 库级 deprecation warning。
  • 少量文案润色、细节间距、训练面板信息密度。
  • 更多 voice 候选和角色人格映射。

资源消耗验收线

你已经遇到过后台有声书生成导致 CPU 温度超过 75°C 的情况，因此本阶段验收必须把资源状态作为 gate。 如果机器正在跑 TTS 批处理、视频渲染、模型训练或大文件转码，不做语音链路压力测试。

阈值建议

  • 正常 温度低于 75°C，页面交互流畅，可继续。
  • 谨慎 75-80°C，减少并发，只跑短脚本。
  • 停止 超过 80°C 持续 3 分钟，暂停测试。
  • 立即停止 超过 85°C 或明显卡顿/风扇异常。

缺陷分级

P0 · 阻断验收

核心链路断、界面被挡、无法恢复、资源失控、页面崩溃。

P1 · 必须进入下轮修复

体验明显违和但不阻断：voice 不合适、反馈不清晰、移动端细节粗糙。

P2 · 可排期

精修项：动效、视觉细节、更多角色、更多训练维度。

验收输出物

  • 命令输出：npm run typecheck、npm test。
  • 截图：桌面首屏、移动首屏、移动 Training 抽屉。
  • 浏览器控制台：error/warn 摘要，区分已知 Live2D warning 与新增错误。
  • 语音记录：所用 avatar、voice id、TTS 是否有声、口型是否同步。
  • 资源记录：测试前温度、测试中峰值、是否有后台重负载进程。

判定口径：当前阶段通过 = 可以进入“真正实时音频/WebRTC 或 Realtime API 技术验证”阶段； 当前阶段不通过 = 先修复 P0，再讨论实时化。
